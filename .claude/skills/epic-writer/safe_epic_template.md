# SAFe Epic Template Structure

## Required Components

### 1. Epic Name
A clear, concise name that describes the epic (typically 3-7 words).

### 2. Epic Description
A brief overview of what the epic is about (2-4 sentences).

### 3. Target Audience
**Who** is this for? Identify the primary users, stakeholders, or customer segments.

### 4. Problem Statement / Need
**What** problem are we solving? What need exists? Include:
- Current pain points
- Why this matters now
- Impact of not solving this

### 5. Why Now?
**Why** is this important to do now? What makes this timely or urgent? Include:
- Market conditions or competitive pressures
- Strategic timing considerations
- Risk of not acting now
- Window of opportunity

### 6. Proposed Solution / Core Features
**What** should this do? Describe the high-level solution and core capabilities:
- Key features or capabilities (typically 3-7 core features)
- How it addresses the problem
- High-level approach
- What makes this solution unique or valuable

### 7. Business Outcome Hypothesis
A statement that follows the format:
"We believe that [doing this] for [these people] will achieve [this outcome]. We will know we are successful when we see [this measurable signal]."

Or alternatively:
"If [we do this], then [this outcome will occur], because [this reasoning]."

### 8. Leading Indicators / Success Metrics
Specific, measurable metrics that will indicate success:
- Quantitative metrics (e.g., % increase, time reduction, cost savings)
- Target values or ranges
- Timeframe for measurement

### 9. Non-Functional Requirements (NFRs)
If applicable, include requirements for:
- Performance
- Scalability
- Security
- Usability
- Compliance

### 10. In Scope / Out of Scope (Optional but recommended)
Clarify boundaries:
- **In Scope**: What this epic includes
- **Out of Scope**: What this epic explicitly does not include

### 11. Dependencies (if known)
- Technical dependencies
- Team dependencies
- External dependencies

### 12. Estimated Timeline / Effort (if available)
- Rough estimate of timeline
- Number of PIs (Program Increments) or quarters

## Formatting Guidelines

- Use clear, business-focused language
- Avoid technical jargon unless necessary
- Be specific about outcomes and metrics
- Focus on value delivery
- Keep it concise but comprehensive
